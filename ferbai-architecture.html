Feature Extraction Module
Pure Python functions — co-located in the same DO container
🌊 Runs inside DO App — no extra infra
Branch A: Claim Extraction
  • Parse transcript → sentence-level claims
  • Tag claims as verifiable (factual) vs. opinion
  • Use spaCy NER + rule-based filters for speed
  • Output: [{claim, sentence_idx, verifiable}]
Branch B: Behavioral Signals
  • Detect rewatch events (segment replay)
  • Detect hesitation (long pause before response)
  • Drawing complexity score (stroke count, coverage)
  • Output: {rewatch_rate, hesitation_ms, drawing_score}
Implementation
  • Single feature_extraction.py module
  • Call from FastAPI background task (non-blocking)
  • No GPU needed — pure CPU logic, runs fast on DO
  • Unit testable independently
Output Contract
  • FeatureBundle Pydantic model
  • Passed directly to both LLM routers
  • Serialized to MongoDB on verdict save
  • Used by swarm to generate varied test signals
⚡ Keep extraction under 500ms. Use async where possible. This is in the critical path before the LLM calls — latency here multiplies across 50 swarm sessions.

Agreement Check → Verdict Engine
Consensus logic — runs in DO container, no extra infra
🌊 Runs inside DO App container
Agreement Logic
  • Both models agree → verdict confidence HIGH
  • Models disagree on any claim → flag for review
  • Score delta >0.3 → escalate to "uncertain" tier
  • Behavioral signals amplify/dampen confidence
Verdict Schema
  • overall_score: 0.0–1.0
  • confidence: low / medium / high
  • flagged_claims: [{claim, reason}]
  • behavioral_notes: {rewatch, hesitation}
Self-Improvement Hook
  • Store disagreement cases as training seeds
  • Build feedback loop: flag → human/LLM label → retrain prompt
  • This is your Continual Learning / Self-Improvement Stack story for judges
Performance
  • Pure Python logic — runs in microseconds
  • No I/O here, only after this (MongoDB write)
  • Produces final VerdictDocument Pydantic model
🎯 Judge pitch: "Two independent LLMs evaluate each session. When they disagree, FerbAI learns — those cases become the training signal for continuous prompt improvement." This hits the Continual Learning theme directly.

Synthetic Student Swarm
50+ Gemma 4 personas running in parallel on DO workers
🌊 DO Droplets / Workers — this is where DO scales
Persona Architecture
  • Gemma 4 on-device (lightweight, edge-ready)
  • Each persona = unique learning profile (confused, advanced, ELL)
  • Personas defined in AGENTS.md + SKILL.md
  • Stored in MongoDB personas collection
DO Scaling
  • Spin up N DO worker processes (or Droplets)
  • Each worker runs one persona agent loop
  • Workers POST to /sessions FastAPI endpoint
  • DO App auto-scales — demo with 50 concurrent sessions
What Swarm Generates
  • Realistic transcripts with deliberate hallucinations
  • Varied behavioral signals (some confused, some fast)
  • Drawing sequences (simple SVG paths)
  • Ground truth labels for evaluation accuracy scoring
Self-Improvement Role
  • Run swarm → collect verdicts → check accuracy vs. ground truth
  • Feed errors back → refine system prompts automatically
  • Run again → measure improvement
  • This is your live recursive self-improvement demo loop
🚀 Killer demo move: Live during judging, run 20 swarm sessions in real-time, show MongoDB filling with verdicts, then show the accuracy improving after one prompt-refinement loop. This is recursive intelligence made visual.
